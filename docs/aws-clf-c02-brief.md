## 1) Who the exam is for

AWS describes the target candidate as someone with **up to 6 months of exposure** to AWS Cloud design, implementation, or operations, often someone early in an AWS career or someone who works alongside cloud teams. The expected knowledge areas are AWS Cloud concepts, security and compliance, core AWS services, and AWS Cloud economics. AWS also explicitly says some tasks are **out of scope for the target candidate**, including coding, cloud architecture design, troubleshooting, implementation, and load/performance testing. ([AWS Documentation][1])

## 2) Exam format and scoring

The guide says the exam uses **multiple-choice** and **multiple-response** questions. There are **50 scored questions** plus **15 unscored questions** used for future exam evaluation, and those unscored items are not identified during the test. Unanswered questions are marked incorrect, and there is **no penalty for guessing**. Results are reported on a **100–1,000 scaled score**, and the **passing score is 700**. AWS also notes that section-level performance feedback is informational only because the exam uses a **compensatory scoring model**, so you do not need to pass each section individually to pass overall. ([AWS Documentation][1])

## 3) The four exam domains and what they really mean

AWS breaks the exam into four weighted domains: **Cloud Concepts (24%)**, **Security and Compliance (30%)**, **Cloud Technology and Services (34%)**, and **Billing, Pricing, and Support (12%)**. Since Domain 3 carries the highest weight, followed by Domain 2, the strongest study emphasis should go there first; that study order is an inference from AWS’s published weights. ([AWS Documentation][1])

### Domain 1: Cloud Concepts — 24%

This domain is about understanding **why cloud matters**. AWS expects you to know the AWS Cloud value proposition, the benefits of global infrastructure, and the advantages of **high availability, elasticity, and agility**. It also expects familiarity with the **AWS Well-Architected Framework** and its pillars, migration strategies and the **AWS Cloud Adoption Framework (AWS CAF)**, plus core cloud economics such as **fixed vs. variable costs, on-premises cost comparisons, BYOL vs. license-included models, rightsizing, automation, and economies of scale**. In plain terms, this section tests whether you understand the business and operating logic behind moving to AWS. ([AWS Documentation][2])

### Domain 2: Security and Compliance — 30%

This is the security foundation section. AWS expects you to understand the **shared responsibility model**, including what AWS secures, what the customer secures, and how that boundary changes across services such as EC2, RDS, and Lambda. You also need familiarity with **compliance and governance**, where to find compliance information such as **AWS Artifact**, logging and auditing resources, encryption in transit and at rest, and security services such as **Amazon GuardDuty, AWS Security Hub, AWS Shield, AWS WAF, Amazon Inspector, AWS CloudTrail, AWS Config, and Amazon CloudWatch**. Identity and access is central too: you need to know **IAM, IAM Identity Center, least privilege, MFA, root user protection, groups/users/policies, roles, and credential handling**. ([AWS Documentation][3])

### Domain 3: Cloud Technology and Services — 34%

This is the biggest section and the most service-oriented. It covers how AWS is accessed and operated through the **Management Console, APIs, SDKs, CLI, and infrastructure as code**, and when to use repeatable processes over one-off actions. It also tests AWS global infrastructure: **Regions, Availability Zones, edge locations, high availability, multi-AZ design, and when multi-Region matters** for disaster recovery, business continuity, latency, or data sovereignty. Then it moves through the major service families: compute, databases, networking, storage, analytics, AI/ML, and several supporting categories. You need to recognize when services fit a use case rather than memorize deep implementation detail. ([AWS Documentation][4])

For **compute**, AWS expects you to distinguish EC2 instance families, containers such as **ECS and EKS**, serverless compute such as **Fargate and Lambda**, auto scaling, and the purpose of load balancers. For **databases**, you should know the difference between **managed databases vs. self-hosted on EC2**, along with relational options like **RDS and Aurora**, NoSQL like **DynamoDB**, in-memory databases like **ElastiCache**, and migration tools such as **AWS DMS and AWS SCT**. For **networking**, the exam expects basic understanding of **VPC components, security groups, network ACLs, Route 53, VPN, and Direct Connect**. For **storage**, focus on **S3 and its storage classes, object vs. block vs. file storage, EBS, EFS, FSx, Storage Gateway, lifecycle policies, and AWS Backup**. ([AWS Documentation][4])

AWS also explicitly includes **AI/ML and analytics** in this domain. You should be able to identify what services like **Amazon SageMaker AI, Amazon Lex, Amazon Kendra, Athena, Kinesis, AWS Glue, and QuickSight** are generally used for. Domain 3 also reaches into other in-scope categories such as **EventBridge, SNS, SQS, Step Functions, Amazon Connect, SES, AWS Support, CodeBuild, CodePipeline, X-Ray, AppStream 2.0, WorkSpaces, Amplify, AppSync, and AWS IoT Core**. ([AWS Documentation][4])

### Domain 4: Billing, Pricing, and Support — 12%

This domain is smaller, but it is very testable because AWS names many specific billing and support concepts. You need to compare **pricing models** such as On-Demand, Reserved Instances, Spot, Savings Plans, Dedicated Hosts, Dedicated Instances, and Capacity Reservations, plus understand storage pricing and **data transfer cost patterns**. AWS also expects familiarity with **AWS Budgets, Cost Explorer, AWS Pricing Calculator, consolidated billing in AWS Organizations, cost allocation tags, and the Cost and Usage Report**. On the support side, know **AWS Support plans**, AWS documentation and whitepapers, **AWS Prescriptive Guidance, AWS Knowledge Center, AWS re:Post**, the role of **Trusted Advisor, AWS Health Dashboard, the AWS Health API**, AWS Marketplace, the **AWS Partner Network**, and where **AWS Professional Services** and **Solutions Architects** fit. ([AWS Documentation][5])

## 4) Technologies and concepts list

AWS includes a separate “Technologies and Concepts” page and says it is **non-exhaustive** and **subject to change**. It highlights broad concepts that may appear anywhere on the exam, including **APIs, cloud migration benefits, AWS CAF, AWS Compliance, compute, cost management, databases, EC2 purchasing models, global infrastructure, IaC, AWS Knowledge Center, machine learning, management and governance, migration and data transfer, network services, AWS Partner Network, AWS Prescriptive Guidance, AWS Pricing Calculator, AWS Professional Services, AWS re:Post, AWS SDKs, security, the Security Blog, the shared responsibility model, storage, Support Center, Support plans, and the Well-Architected Framework**. This is a useful clue that the exam is not only about service names; it also tests basic cloud ideas and AWS ecosystem resources. ([AWS Documentation][6])

## 5) In-scope AWS services

AWS provides a separate **In-Scope AWS Services** reference and says it is also **non-exhaustive** and **subject to change**. The categories include **Analytics, Application Integration, Business Applications, Cloud Financial Management, Compute, Containers, Customer Enablement, Database, Developer Tools, End User Computing, Frontend Web and Mobile, IoT, Machine Learning, Management and Governance, Migration and Transfer, Networking and Content Delivery, Security/Identity/Compliance, Serverless, and Storage**. Representative services named by AWS include **Athena, Redshift, QuickSight, EventBridge, SNS, SQS, Step Functions, EC2, Elastic Beanstalk, Lightsail, ECS, EKS, Aurora, DynamoDB, RDS, AWS CLI, Amplify, IoT Core, SageMaker AI, CloudFormation, CloudTrail, CloudWatch, Config, Organizations, Trusted Advisor, Application Migration Service, DMS, Snow Family, API Gateway, CloudFront, Route 53, VPC, IAM, IAM Identity Center, KMS, GuardDuty, Security Hub, Shield, WAF, Lambda, Fargate, S3, EBS, EFS, FSx, Glacier, Backup, and Storage Gateway**. ([AWS Documentation][7])

## 6) Out-of-scope AWS services

AWS also publishes an **Out-of-Scope AWS Services** list, again marked **non-exhaustive** and **subject to change**. Examples AWS names include **AppFlow, Clean Rooms, Data Exchange, DataZone, MSK, App Runner, Copilot, Billing Conductor, Keyspaces, MemoryDB, AppConfig, CodeArtifact, CodeDeploy, CodeGuru, CloudShell, Device Farm, GameLift, Greengrass, Fraud Detector, Personalize, Chatbot, Launch Wizard, Elemental media services, Transfer Family, Cloud Map, VPC Lattice, Network Firewall, RoboMaker, and FSx for Lustre**. The practical takeaway is that these are lower priority for CLF-C02 prep compared with the in-scope list. ([AWS Documentation][8])

## 7) Best way to interpret the guide

The guide itself says it includes **weightings, content domains, and task statements**, but it does **not** provide a complete list of all exam content. So the best way to use it is as a blueprint: study the four domains by weight, learn the “why” behind AWS Cloud concepts, get comfortable with major security and identity ideas, recognize the primary service categories and common use cases, and use the in-scope/out-of-scope lists to decide where to spend your time. ([AWS Documentation][1])
